# CLAUDE.md — Next.js 15 + SQLite SaaS Project

> Opinionated Claude Code rules for a production-ready SaaS built with Next.js 15 (App Router) and SQLite (better-sqlite3 / Turso).

---

## Stack & Versions

- **Framework**: Next.js 15 (App Router)
- **Language**: TypeScript 5.x (strict mode)
- **Database**: better-sqlite3 (local/dev) / Turso (production)
- **ORM**: Prisma 5.x (schema-first migrations)
- **Styling**: Tailwind CSS 4.x
- **Auth**: NextAuth.js v5 / Clerk (project decides at init)
- **Package Manager**: pnpm (enforce via .npmrc with package-manager-lockfile-check=true)
- **Runtime**: Node.js 20 LTS+

**Why**: Next.js 15 App Router enforces a server-first architecture. SQLite via Prisma gives us zero-config local dev with zero migration friction in CI. pnpm enforces deterministic dependency trees.

## Database & Migration Rules

1. **Always use Prisma client** — never raw SQL in components or API handlers.
2. **Schema lives in prisma/schema.prisma** — that is the single source of truth. Types are generated from it.
3. **Migrations are committed** — pnpm db:migrate:save generates SQL that must be committed alongside the schema change.
4. **Use soft deletes** (deletedAt: DateTime?) — never DELETE from production tables.
5. **SQLite-specific**:
   - Use etter-sqlite3 for dev (synchronous, fast, zero-config).
   - Use @libsql/client for production Turso (async, edge-compatible).
   - Abstract the difference behind a db interface in lib/db/client.ts.
6. **Foreign keys are ON** — PRAGMA foreign_keys = ON in every connection. No orphaned records.

**Why**: Prisma gives type-safe queries. Soft deletes enable audit trails. SQLite FK enforcement prevents data corruption. The client abstraction lets us swap backends without touching business logic.

## Component Patterns

1. **Server Components by default** — every page.tsx and layout.tsx is a Server Component. Only add "use client" when you need hooks, events, or state.
2. **Presentational components are pure** — no data fetching, no side effects. Receive props, return JSX.
3. **Container components handle data** — fetch, transform, pass to presentational children.
4. **Forms use React Hook Form + Zod** — validation schema in lib/validation.ts, form logic in components.
5. **No prop drilling beyond 2 levels** — use Context or Zustand for shared state.
6. **Naming**:
   - Components: PascalCase (UserProfile.tsx)
   - Hooks: camelCase with use prefix (useAuth.ts)
   - Utilities: camelCase (ormatCurrency.ts)
   - Types: PascalCase (UserProfileProps)
   - Constants: UPPER_SNAKE_CASE (MAX_UPLOAD_SIZE)

**Why**: Server-first reduces client bundle. Containers decouple data from rendering. Strict naming prevents ambiguity.

## Anti-Patterns (What We DON'T Do)

1. **No etch() in Server Components for data** — use Prisma queries or lib/db/queries/.
2. **No ny types** — strict TypeScript is non-negotiable.
3. **No inline styles** — use Tailwind utility classes. CSS modules only as last resort.
4. **No direct console.log in production** — use a logger (lib/utils.ts → logger.info/warn/error).
5. **No unvalidated user input** — every external input passes through a Zod schema.
6. **No useEffect for data fetching** — rely on Server Components or SWR/React Query for client-side.
7. **No monolithic page.tsx** — split into components. Max 200 lines per file.
8. **No environment variables without Zod validation** — all config validated at startup.

**Why**: These patterns cause bugs, security issues, and maintenance debt. Each rule exists because we've seen the alternative fail in production.

## Security Rules

1. **All API routes must validate input** — Zod schema required for every POST/PATCH body.
2. **Rate limiting on auth endpoints** — use @upstash/ratelimit or equivalent.
3. **CSRF protection** — cookie-based sessions only. No JWT in localStorage.
4. **SQL injection prevention** — Prisma parameterized queries only. Never concatenate user input into SQL strings.
5. **File uploads** — validate MIME type AND extension on server. Store in S3-compatible storage, not local filesystem.
6. **Headers** — set X-Content-Type-Options: nosniff, X-Frame-Options: DENY, Strict-Transport-Security in production.

## Testing Strategy

1. **Unit tests** (	ests/unit/) — test pure functions, hooks, validators. Fast, isolated.
2. **Integration tests** (	ests/integration/) — test API routes with a real SQLite database.
3. **No E2E tests in initial sprint** — add Playwright tests only after core flows are stable.
4. **Test data** lives in 	ests/fixtures/ — seeded via pnpm db:seed.
5. **Coverage threshold**: 80% minimum for lib/ and 	ests/unit/.

## Performance Rules

1. **Image optimization** — always use 
ext/image. Never <img> tags directly.
2. **Font optimization** — use 
ext/font for all custom fonts. Subset for production.
3. **Route-level code splitting** — heavy components lazy-loaded via 
ext/dynamic.
4. **SQLite is synchronous** — long-running queries block the event loop. Use Promise.all() for parallel queries. Never await sequential independent queries.
5. **Cache aggressively** — evalidateTag and evalidatePath for data that changes infrequently. Default cache: 60s.

## When in Doubt

- **Server over Client** — if it doesn't need interactivity, it's a Server Component.
- **Prisma over Raw SQL** — type safety is worth the abstraction layer.
- **Explicit over Implicit** — name things clearly, add comments for WHY not WHAT.
- **Fail Fast** — validate input early, reject bad data immediately.
- **SQLite Simplicity** — don't overcomplicate. SQLite is fast enough for most SaaS workloads under 1M rows. Scale to PostgreSQL only when benchmarks prove SQLite is the bottleneck.
